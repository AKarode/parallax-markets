# Phase 10.1: Code Review — Multi-Call Claude Ensemble

**Reviewed:** 2026-04-15
**Depth:** standard
**Files Reviewed:** 6

## Summary

Phase 10.1 is well-structured and the core ensemble design is sound. The `ensemble_predict` abstraction cleanly replaces three copies of single-call logic, `WorldState.copy()` correctly solves the mutation safety problem, and the trimmed-mean aggregation is appropriate for n=3.

However, two high-severity bugs exist: (1) the budget tracker is not checked before firing 3 concurrent API calls, so an already-over-budget run will triple spending before the guard can stop it; (2) `confidence` from the "median" parsed response can exceed 1.0 after the 0.5 penalization is skipped — the penalized value is never clamped before being passed to `PredictionOutput`, which will raise a Pydantic `ValidationError` at runtime if Claude returns `confidence > 1.0` in an unstable ensemble (the penalization path is fine, but the non-penalized path is not validated). Three medium-severity issues cover partial ensemble failure masking, `_iter_cells` accessing a private attribute, and an unchecked `magnitude_range` length. Two low-severity issues cover the duplicate `events[:10]` / `events[:20]` slice discrepancy and a missing `is_empty` guard in `compute_ensemble`.

## High Severity

### H-01: Budget not checked before 3 concurrent LLM calls

**File:** `backend/src/parallax/prediction/ensemble.py:106`

**Issue:** `ensemble_predict` fires all 3 `asyncio.gather` calls unconditionally, without consulting `budget.is_over_budget()` first. `BudgetTracker.is_over_budget()` exists precisely for this guard. If the pipeline runs at or near the $20/day cap (e.g., after a previous failed run that already used budget), this function will issue 3 more Anthropic API calls and triple the overspend before the caller can react. With the ensemble now used by all 3 predictors, a single brief run makes 9 concurrent calls total — the exposure is 9x.

**Fix:**
```python
async def ensemble_predict(client, model, prompt, budget, max_tokens=2000) -> dict:
    if budget.is_over_budget():
        raise ValueError("Daily LLM budget exceeded — skipping ensemble call")
    # ... rest of function
```

### H-02: `confidence` value not clamped before PredictionOutput construction

**File:** `backend/src/parallax/prediction/oil_price.py:141-143`, same pattern in `ceasefire.py:114-116` and `hormuz.py:116-118`

**Issue:** Claude is instructed to return `confidence` as a float 0–1, but LLMs sometimes return values slightly outside this range (e.g., `1.0` rounded up to `1.01`, or negative when expressing "very low confidence"). The Pydantic `clamp_probability` validator on `PredictionOutput` raises `ValueError` rather than clamping — it does NOT clamp, it rejects. So if Claude returns `confidence=1.01`, the entire predictor raises a `ValidationError` and the brief run fails with no output. This is especially likely after the ensemble penalization path: `confidence *= 0.5` is safe, but the non-penalized path accepts whatever Claude returns raw.

```python
# In schemas.py line 33-34:
if not 0.0 <= v <= 1.0:
    raise ValueError(...)  # raises — does NOT clamp
```

**Fix — option A (preferred): clamp in the validator:**
```python
@field_validator("probability", "confidence")
@classmethod
def clamp_probability(cls, v: float) -> float:
    return max(0.0, min(1.0, float(v)))
```

**Fix — option B: clamp at call site in each predictor:**
```python
confidence = max(0.0, min(1.0, parsed.get("confidence", 0.5)))
if ensemble["is_unstable"]:
    confidence = max(0.0, min(1.0, confidence * 0.5))
```

Option A is safer — it protects `probability` too, which has the same exposure.

## Medium Severity

### M-01: Partial ensemble failure silently degrades to single call with no warning to caller

**File:** `backend/src/parallax/prediction/ensemble.py:114-134`

**Issue:** If 2 out of 3 calls fail (API timeout, rate limit, etc.), the function succeeds with `call_count=1` and silently returns a single-call result. The caller receives `ensemble_std_dev=0.0` and `ensemble_is_unstable=False`, which will look like a high-confidence stable signal — the opposite of what partial failure should signal. The `call_count` field is returned but none of the callers (`oil_price.py`, `ceasefire.py`, `hormuz.py`) check it.

**Fix:** Raise (or at minimum emit a warning) when `call_count < len(ENSEMBLE_TEMPERATURES)`, and treat degraded calls as unstable:

```python
if len(all_parsed) < len(ENSEMBLE_TEMPERATURES):
    logger.warning(
        "Ensemble degraded: only %d/%d calls succeeded",
        len(all_parsed), len(ENSEMBLE_TEMPERATURES),
    )
    # Force instability flag when operating on fewer calls than designed
    if ensemble_result and len(all_parsed) < 2:
        ensemble_result["is_unstable"] = True
```

Alternatively, each predictor should check `result["call_count"] < 3` and penalize confidence accordingly.

### M-02: `_iter_cells` and `_get_hormuz_status` access private `_cells` attribute directly

**File:** `backend/src/parallax/prediction/oil_price.py:164-167`, `backend/src/parallax/prediction/hormuz.py:146`

**Issue:** Both methods iterate `ws._cells.keys()` directly, bypassing `WorldState`'s public interface. `_cells` is a private attribute (underscore convention). `WorldState.snapshot()` already provides a public iteration path. If `WorldState` internals change (e.g., sharding large cell sets), these callers will break silently. Additionally, `_estimate_recovery` in `hormuz.py` at line 170 also iterates `ws._cells` directly for the sum.

**Fix:** Use the existing public interface:
```python
# Instead of: for cell_id in list(ws._cells.keys())
# Use: for cell in ws.snapshot()  — already returns dicts with all fields
```

Or add a thin public iterator to `WorldState`:
```python
def iter_cells(self) -> list[dict]:
    return self.snapshot()
```

### M-03: `magnitude_range` not validated for length before indexing

**File:** `backend/src/parallax/prediction/schemas.py:18`

**Issue:** `magnitude_range: list[float]` has no length constraint. The callers in `oil_price.py`, `ceasefire.py`, and `hormuz.py` pass `parsed.get("magnitude_range", [0.0, 1.0])` directly. If Claude returns a malformed `magnitude_range` (e.g., a single-element list, an empty list, or 3 elements), any downstream code that indexes `magnitude_range[0]` or `magnitude_range[1]` will raise `IndexError`. This is a real LLM failure mode — the model occasionally returns `[25]` instead of `[25, 45]`.

**Fix — in schemas.py:**
```python
from pydantic import field_validator

@field_validator("magnitude_range")
@classmethod
def validate_magnitude_range(cls, v: list[float]) -> list[float]:
    if len(v) != 2:
        raise ValueError(f"magnitude_range must have exactly 2 elements, got {len(v)}: {v}")
    return v
```

Or defensively pad at the call site:
```python
raw_range = parsed.get("magnitude_range", [0.0, 1.0])
magnitude_range = raw_range[:2] if len(raw_range) >= 2 else [raw_range[0], raw_range[0]] if raw_range else [0.0, 1.0]
```

## Low Severity

### L-01: Slice discrepancy between `_format_events` caller and implementation

**File:** `backend/src/parallax/prediction/oil_price.py:115`, `backend/src/parallax/prediction/ceasefire.py:93`, `backend/src/parallax/prediction/hormuz.py:96`

**Issue:** All three callers pass `recent_events[:10]` to their respective `_format_events` methods, but `_format_events` in all three predictors internally slices again with `events[:20]`. The outer slice to 10 makes the inner slice to 20 dead code — it will never do additional filtering beyond what the caller already passed. This creates a false impression that up to 20 events are considered when actually at most 10 are. If the intent is to feed 20 events, remove the outer `[:10]`; if 10 is the right limit, remove the inner `[:20]`.

**Fix:** Make the limit explicit in one place:
```python
# In the predictor call sites, remove the outer slice:
events_summary = self._format_events(recent_events)
# And keep the [:20] inside _format_events as the canonical limit.
```

### L-02: `compute_ensemble` does not handle empty input list

**File:** `backend/src/parallax/prediction/ensemble.py:53-72`

**Issue:** `compute_ensemble([])` returns `{"probability": 0.5, ...}` via the `probabilities[0] if probabilities else 0.5` guard, which is correct. However, `len(probabilities) < 2` at line 59 will be True for the empty list AND for the single-element list, meaning the 0.5 default path is only reached inside the `if` block's own ternary — this is fine but the comment in `trimmed_mean` says "n<=2 returns arithmetic mean" while `statistics.mean([])` raises `StatisticsError`. If `compute_ensemble` is ever called directly (not through `ensemble_predict`), an empty list will return `0.5` correctly, but `trimmed_mean([])` would crash. The current flow never reaches `trimmed_mean` with an empty list, but the safety margin is thin.

**Fix:** Add a guard in `trimmed_mean`:
```python
def trimmed_mean(values: list[float]) -> float:
    if not values:
        return 0.5
    if len(values) <= 2:
        return statistics.mean(values)
    ...
```
