# Phase 10.1: Multi-Call Claude Ensemble - Research

**Researched:** 2026-04-15
**Domain:** LLM ensemble prediction, async Python, state mutation safety
**Confidence:** HIGH

## Summary

This phase adds ensemble prediction by making 3 independent Claude calls per predictor with varied temperature, aggregating via trimmed mean, and flagging unstable estimates. It also fixes a WorldState mutation bug where `apply_blockade()` modifies shared state in-place during oil price prediction.

The implementation is straightforward: each predictor's `predict()` method already makes a single LLM call and parses JSON. The ensemble wraps this with 3 concurrent calls via `asyncio.gather`, aggregates the parsed probabilities, and enriches the `PredictionOutput` with per-call detail. The WorldState bug fix requires a `copy()` method on WorldState and using it before cascade operations.

**Primary recommendation:** Build a shared `ensemble_predict()` utility function that any predictor can call, rather than duplicating ensemble logic in each predictor class. Use `copy.deepcopy()` on WorldState before any mutating cascade operations.

**Critical budget finding:** All 3 predictors currently use `claude-opus-4-20250514`, NOT Sonnet. At Opus 4 pricing ($15/$75 per million input/output tokens), 9 calls per run costs approximately $0.63/run (not $0.063). The phase description assumes Sonnet pricing. The implementation must either: (a) switch to Sonnet 4 for ensemble calls (recommended -- $0.063/run), or (b) accept the higher Opus cost within the $20/day budget (2 runs/day = $1.26/day, still under cap). ### Pattern 1: Ensemble Predict Utility

**What:** A shared async function that takes a prompt and client, makes N calls with varied temperatures, and returns aggregated results.
**When to use:** Every predictor's predict() method delegates the LLM call to this utility.

```python
# backend/src/parallax/prediction/ensemble.py
import asyncio
import statistics
from typing import Any

ENSEMBLE_TEMPERATURES = [0.3, 0.5, 0.7]
INSTABILITY_THRESHOLD = 0.10  # 10 percentage points


async def ensemble_predict(
    client: Any,
    model: str,
    prompt: str,
    max_tokens: int = 2000,
    n_calls: int = 3,
) -> dict:
    """Make N independent Claude calls and aggregate probabilities.

    Returns dict with:
      - parsed_responses: list of parsed JSON dicts
      - probability: trimmed mean of individual probabilities
      - std_dev: standard deviation across calls
      - is_unstable: True if std_dev > INSTABILITY_THRESHOLD
      - individual_probabilities: list of floats
    """
    temperatures = ENSEMBLE_TEMPERATURES[:n_calls]

    async def _single_call(temperature: float) -> dict:
        response = await client.messages.create(
            model=model,
            max_tokens=max_tokens,
            temperature=temperature,
            messages=[{"role": "user", "content": prompt}],
        )
        return response

    responses = await asyncio.gather(*[_single_call(t) for t in temperatures])
    return responses  # Caller parses and aggregates
```

### Anti-Patterns to Avoid

- **Duplicating ensemble logic in each predictor:** All 3 predictors have near-identical LLM call + parse patterns. Extract the common pattern into `ensemble.py` rather than copy-pasting 3x.
- **Using shallow copy for WorldState:** `CellState` is a mutable dataclass. Shallow copy would share CellState references. Must use `deepcopy`.
- **Averaging all 3 without trimmed mean:** With only 3 values, trimmed mean (dropping highest and lowest) reduces to the median. This is desirable -- it discards outlier calls.
- **Modifying temperature on the existing single call:** The point is 3 independent calls. Don't change the existing call's temperature; add 2 more calls alongside it.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Standard deviation | Custom variance calc | `statistics.stdev()` | Edge cases with n=1, n=2 handled |
| Deep copy of state | Manual field-by-field clone | `copy.deepcopy()` | WorldState has nested dict of mutable CellState dataclasses |
| Concurrent API calls | Sequential loop | `asyncio.gather()` | Already used in brief.py for 3 predictors; pattern is established |

## Common Pitfalls

### Pitfall 1: Trimmed Mean with 3 Values Is Just Median
**What goes wrong:** With only 3 values, 20% trim from each side drops 0.6 values, which rounds to dropping 1 from each end -- leaving just the middle value (median).
**Why it happens:** Trimmed mean is designed for larger samples.
**How to avoid:** This is actually fine for this use case. The median of 3 is robust to a single outlier. Document that "trimmed mean with n=3" equals "median" so future developers aren't confused.
**Warning signs:** None -- this is the correct behavior.

### Pitfall 2: Budget Tracking Must Count All 3 Calls
**What goes wrong:** BudgetTracker.record() is called once instead of 3 times, understating costs by 3x.
**Why it happens:** Easy to forget when refactoring from single-call to ensemble.
**How to avoid:** Record budget for each response in the ensemble loop, not just the final aggregated result.
**Warning signs:** Budget stats show 3 calls instead of 9 per run.

### Pitfall 3: JSON Parse Failures in Ensemble
**What goes wrong:** One of 3 calls returns malformed JSON, crashing the entire prediction.
**Why it happens:** Temperature 0.7 is more likely to produce formatting errors.
**How to avoid:** Parse each call independently. If 1 of 3 fails, aggregate from the remaining 2 (or 1). Only fail if all 3 fail to parse.
**Warning signs:** Occasional crashes at higher temperature settings.

### Pitfall 4: WorldState Mutation Already Happened in Previous Runs
**What goes wrong:** The mutation bug has been present since Phase 10. Historical data in DuckDB may reflect corrupted cascade calculations.
**Why it happens:** oil_price.predict() calls apply_blockade() which mutates the shared WorldState before hormuz.predict() runs.
**How to avoid:** Fix forward. Historical predictions are already stored. Note in commit message that prior cascade numbers may have been affected.
**Warning signs:** Looking at brief.py line 492-500: `asyncio.gather` runs all 3 predictors concurrently. Since oil_pred and hormuz_pred share the same `world_state` reference AND both run concurrently via gather, the mutation is a race condition, not a guaranteed corruption. But it's still wrong.

### Pitfall 5: Model Cost -- Opus vs Sonnet
**What goes wrong:** Budget math in phase description assumes Sonnet (~$0.007/call). Actual model is Opus 4 (~$0.07/call). 9 calls = $0.63/run, not $0.063.
**Why it happens:** Phase description was written assuming Sonnet without checking codebase.
**How to avoid:** Either switch to Sonnet 4 for ensemble (recommended) or accept higher cost. At 2 runs/day, Opus ensemble = $1.26/day (well under $20 cap but 10x the assumed cost).
**Warning signs:** Budget tracker reports higher-than-expected spend per run.

## Code Examples

### Ensemble Aggregation (trimmed mean with n=3)

```python
# With 3 values, trimmed mean = median
import statistics

def trimmed_mean(values: list[float]) -> float:
    """Trimmed mean for ensemble probabilities. With n=3, equals median."""
    if len(values) <= 2:
        return statistics.mean(values)
    sorted_vals = sorted(values)
    # Drop lowest and highest, take mean of remainder
    trimmed = sorted_vals[1:-1]
    return statistics.mean(trimmed)

def compute_ensemble(probabilities: list[float]) -> dict:
    """Aggregate ensemble probabilities with instability detection."""
    if len(probabilities) < 2:
        return {
            "probability": probabilities[0] if probabilities else 0.5,
            "std_dev": 0.0,
            "is_unstable": False,
            "individual_probabilities": probabilities,
        }
    return {
        "probability": trimmed_mean(probabilities),
        "std_dev": statistics.stdev(probabilities),
        "is_unstable": statistics.stdev(probabilities) > 0.10,
        "individual_probabilities": probabilities,
    }
```

## Open Questions

1. **Should ensemble calls use Sonnet 4 instead of Opus 4?**
   - What we know: Current predictors use Opus 4. Ensemble triples cost. At Opus pricing, 9 calls/run = ~$0.63. At Sonnet pricing, ~$0.063.
   - What's unclear: Whether prediction quality degrades meaningfully with Sonnet vs Opus for this use case.
   - Recommendation: Switch to Sonnet 4 (`claude-sonnet-4-20250514`) for ensemble calls. The ensemble's diversity benefit compensates for per-call quality reduction. If quality drops, can upgrade back to Opus (budget allows it).

2. **How should LOW_CONFIDENCE flag affect downstream signal generation?**
   - What we know: ENS-02 says "signal strength is downgraded." Current signal generation uses `effective_edge`.
   - What's unclear: Should LOW_CONFIDENCE halve the confidence score, reduce effective_edge, or add a separate gate?
   - Recommendation: Multiply confidence by 0.5 when unstable. This flows through to MappingPolicy's edge calculation naturally.

3. **Should ensemble reasoning include all 3 individual reasonings?**
   - What we know: ENS-01 says "reasoning output includes all 3 individual probabilities and aggregation method."
   - What's unclear: Including 3 full reasoning blocks (500-1000 words each) would make reasoning very long.
   - Recommendation: Include the median call's full reasoning plus a summary line listing all 3 probabilities and the aggregation method. Don't include all 3 full reasoning texts.
