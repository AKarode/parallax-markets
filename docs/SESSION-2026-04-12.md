## Key Design Decisions

1. **Crisis context as static text, not dynamic retrieval.** The timeline is a Python string constant, not fetched from an API. This is intentional — it's updated manually as events unfold, giving full control over what the model sees.

2. **Backtest uses date-limited context.** For each historical day, the model only sees events up to that date. This prevents lookahead bias. Valid because Claude's training cutoff (Aug 2025) predates the entire crisis (Feb 2026+).

3. **Oil deconfliction at signal level, not mapping level.** The deconfliction happens after signals are generated, not inside MappingPolicy. This keeps the audit trail complete — both signals exist in the ledger, but the loser gets downgraded to HOLD.

4. **Cron stops/restarts uvicorn.** DuckDB only allows one writer. Rather than implementing WAL or read-only mode, the cron script kills uvicorn, runs the pipeline, then restarts it via `trap EXIT`. Simple and reliable.

## Setbacks and Fixes

### 1. Cron `python: command not found`
**Problem:** Cron runs at 8am/8pm failed with `python: command not found` since Apr 10. ~36 hours of no data.
**Root cause:** macOS cron has a minimal PATH that doesn't include anaconda.
**Fix:** Script now explicitly resolves Python: venv → anaconda → miniconda → system python3.
**Lesson:** Always use absolute paths in cron scripts.

### 2. DuckDB lock conflict between uvicorn and CLI
**Problem:** CLI pipeline couldn't write to DuckDB while uvicorn held the write lock.
**Root cause:** DuckDB is single-writer. Both the API server and CLI try to open write connections.
**Fix:** Cron script stops uvicorn before pipeline, restarts after via `trap EXIT`.
**Lesson:** DuckDB's single-writer model means you need process coordination for concurrent access.

### 3. API endpoint hardcoded to dry_run
**Problem:** `POST /api/brief/run` always returned mock data even with Kalshi keys loaded.
**Root cause:** Line 252 of main.py: `await run_brief(dry_run=True, ...)` — hardcoded.
**Fix:** Changed to `dry_run=False`.
**Lesson:** Check the obvious things first.

### 4. Resolution checker 404s on event-level tickers
**Problem:** `_check_market_resolution()` failed with 404 on `KXWTIMAX-26DEC31` and `KXWTIMIN-26DEC31`.
**Root cause:** These are event tickers, not market tickers. Kalshi's `/markets/{ticker}` expects contract-level tickers (e.g., `KXWTIMAX-26DEC31-T140`).
**Fix:** Added fallback: try direct lookup, if 404 try `/markets?event_ticker=` to find child markets.
**Lesson:** Kalshi has event tickers (group) vs market tickers (individual contracts). Always handle both.

### 5. Portfolio simulator showing flat $1,000
**Problem:** Simulator never opened positions despite 8 BUY signals in the ledger.
**Root cause:** Two bugs: (a) `entry_price` and `entry_side` were NULL on all legacy signals (pre-executable-price era), (b) weighted ensemble aggregation included HOLD signals with 0 edge, diluting real edges below the 5% threshold.
**Fix:** (a) Backfilled `entry_price` from `market_yes_price`/`market_no_price` and inferred `entry_side` from signal type. (b) Changed aggregation to only include BUY signals in weighted average.
**Lesson:** Simulator was technically working — it correctly rejected signals with no executable price. The data was the problem.

### 6. Models predicting blind — no crisis knowledge
**Problem:** Predictions were generic and incoherent. Ceasefire model predicted "62% ceasefire holding" when mapped to a "formal agreement" contract.
**Root cause:** Claude's training data ends ~Aug 2025. The entire Iran-Hormuz crisis (war, Khamenei assassination, Hormuz closure, ceasefire) started Feb 2026. The models literally didn't know any of this happened.
**Fix:** Created `crisis_context.py` with full timeline from Feb-Apr 2026, injected into every prediction prompt. Rewrote ceasefire prompt to predict "formal agreement" not "ceasefire holding."
**Lesson:** This was the single biggest improvement. Always check what the model actually knows vs what it needs to know.

### 7. Stale test assertions
**Problem:** 7 tests in `test_mapping_policy.py` failing — asserting old discount-based edge math.
**Root cause:** Tests were written before the fair-value estimator system replaced multiplicative proxy discounts. The code evolved but tests didn't.
**Fix:** Updated assertions to test current behavior (confidence_discount=1.0, fair-value estimators active, cost model deducts from edge).
**Lesson:** Pre-existing test debt. The 1 failure in `test_recalibration.py` is also pre-existing and unrelated to this session.

## Known Limitations

1. **Models can't predict diplomatic surprises.** The Apr 7 ceasefire was the backtest's biggest loss. No amount of context fixes this.
2. **Next-day scoring penalizes contrarian bets.** The agreement model may be right at settlement but loses money day-to-day as the hope trade runs.
3. **No multi-day news context.** Models see a single run's headlines. Can't distinguish "new signal" from "day 5 of same story."
4. **Crisis context is manually maintained.** Someone needs to update `crisis_context.py` as events unfold.
5. **Backtest doesn't include fees/slippage.** The -$0.35 P&L would be worse net of costs.
