# Session: April 9, 2026

## Summary

Implemented the first real trading-infrastructure sprint for Parallax.

The repo no longer treats a generic `yes_price` snapshot as an executable trading input. Market snapshots now expose explicit bid/ask semantics, signal generation computes edge against actual entry cost, paper trading records an order/fill/position lifecycle, and trading evaluation uses only real filled paper positions.

## What Changed

### 1. Market Data Normalization

| File | Change |
|------|--------|
| `markets/schemas.py` | Replaced snapshot-centric market model with explicit executable quotes, spreads, timestamps, and optional depth summaries |
| `markets/kalshi.py` | Fixed Kalshi normalization so asks are derived honestly from the opposite-side bid book when needed |
| `markets/polymarket.py` | Added executable CLOB book normalization and labeled derived fallbacks |
| `db/schema.py` | Expanded `market_prices` table with executable quote fields and depth metadata |

Important semantics:

- `best_yes_ask` is the cost to buy YES
- `best_no_ask` is the cost to buy NO
- `yes_price` / `no_price` remain as labeled derived fields for display and reference only
- quote provenance is explicit via `derived_price_kind`

### 2. Executable Edge Computation

| File | Change |
|------|--------|
| `divergence/detector.py` | Computes `BUY_YES` and `BUY_NO` edge from executable asks |
| `contracts/mapping_policy.py` | Evaluates both YES and NO entry paths and chooses the better executable side |
| `contracts/schemas.py` | Mapping results now include entry side, entry price, price kind, and tradeability status |
| `cli/brief.py` | Brief output now labels executable entry vs derived reference pricing |

Behavior changes:

- if no executable entry price exists, the system refuses to trade
- edge is no longer computed from ambiguous `market_probability`
- tradability is explicit instead of implied

### 3. Real Paper-Trade Journal

| File | Change |
|------|--------|
| `scoring/tracker.py` | Replaced fake paper trade rows with `order -> fill -> position` lifecycle handling |
| `scoring/ledger.py` | Signals now store execution status and order/position linkage |
| `db/schema.py` | Added `trade_orders`, `trade_fills`, `trade_positions` tables |
| `main.py` | `/api/trades` now returns journal orders and positions |
| `dashboard/data.py` | Trade/journal queries now surface execution state |

Lifecycle now distinguished explicitly:

- signal
- order attempt
- accepted order
- rejected order
- cancelled order
- fill
- open position
- closed position

Failed execution attempts no longer create fake open trades.

### 4. Traded-Only Evaluation

| File | Change |
|------|--------|
| `scoring/resolution.py` | Settlement now backfills signal-quality outcomes separately from closing traded positions |
| `scoring/report_card.py` | Trading report card uses only actual closed positions |
| `scoring/calibration.py` | Calibration now runs on signal-quality / counterfactual data, clearly labeled |
| `db/schema.py` | Added `signal_quality_evaluation` and `trade_evaluation` views |

New split:

- `signal_quality_evaluation`: resolved signals, including untraded counterfactual analysis
- `trade_evaluation`: only actual filled and later closed paper positions

### 5. Dry-Run and Environment Isolation

| File | Change |
|------|--------|
| `db/runtime.py` | Added explicit runtime/data-environment resolution |
| `cli/brief.py` | Dry-run now isolates storage by default and persists environment labels |
| `main.py` | API health reports environment |
| `prediction_log.py` | Prediction records include environment labels |

Behavior:

- dry runs default to isolated in-memory storage
- demo/live DBs can be separated with `PARALLAX_ENV`
- persisted records carry `data_environment` and execution environment metadata

## Schema Changes

New tables:

- `trade_orders`
- `trade_fills`
- `trade_positions`

New views:

- `signal_quality_evaluation`
- `trade_evaluation`

Expanded tables:

- `market_prices`
- `signal_ledger`
- `prediction_log`

Migration approach:

- `create_tables()` performs additive migration on startup
- legacy fields such as `market_yes_price`, `market_no_price`, `trade_id`, and `realized_pnl` are retained as compatibility aliases, but new execution logic does not use them as trading inputs
